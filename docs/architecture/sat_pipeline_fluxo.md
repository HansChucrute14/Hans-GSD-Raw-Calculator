## 5. Two Usage Categories — V10

### 5.1 "Free" Category

**Flow:**
1. User selects any set of ingredients (1 to N).
2. Build pipeline computes DER → dynamic envelope → builds a_ij matrix.
3. Solver executes cascade (Level 1 → 2 → 3, stops at first feasible).
4. Output includes: `allocations` (if Level 1/2) or `diagnostic_analysis` (if Level 3), `nutrient_results`, `gaps`, `alerts`, `recommended_additions`.
5. Interface displays contextual alerts per `solver_status` and `feeding_recommendation`.

**Displayed alerts:**

|Condition|Message|Severity|
|---|---|---|
|Nutrient above SUL|*"Warning: dangerous excess of {nutrient} at {pct}% above safe limit"*|`danger`|
|Nutrient below minimum|*"{pct}% short of ideal minimum for {nutrient}"*|`warning`|
|Ca:P ratio out of range|*"Ca:P ratio of {value} — outside safe range [1.1, 1.3]"*|`warning`|
|No bone/calcium source|*"Diet without calcium source — DOD risk in growth"*|`danger`|

**`recommended_additions`** is generated by category (`ingredient['category']` from `DB_ingredientes.json`), not by specific ingredient — maps `gap.nutrient_id → category(s) from DB with highest concentration of that nutrient`, ordered by real concentration. Deterministic computation on existing JSONs, no new data.

### 5.2 "Precomputed Recipes" Category

**Concept:** Precomputed Recipes **is not a runtime feature** — it's a precomputed artifact in the same pipeline. Needs nothing new in real time and does not require solver running in the user's browser.

**Generation (build pipeline in `--build-recipes` mode):**
1. `python build_pipeline.py` (delegates to `src/gsd/cli.py:main()`) runs solver in *build* mode, over restricted combinatorial space of N ingredients.
2. Respects minimum category coverage (at least 1 `muscle_meat` + 1 `organ_secreting` + 1 calcium source), following `diet_templates` already in `formulation_rules.json`.
3. Saves result in versioned `recipes_precomputed.json`. Frontend only reads this JSON.

**Ranking by 5+ criteria:**

|Criterion|Weight|How calculated|Rationale|
|---|---|---|---|
|**Criticality-weighted completeness**|0.30|Σ (pct_of_min_achieved × clinical_criticality_weight) / Σ (clinical_criticality_weight)|Not just "how many of 41 hit", but weighted sum. Hitting 2 macronutrients and 5 critical minerals is worth more than hitting 10 desirable amino acids.|
|**Safety margin**|0.25|min(1 - pct_of_sul_nearest) among 8 SULs|Minimum distance to any SUL — larger is better. Direct proxy for "how far from danger" recipe stays even with real batch variation.|
|**Price**|0.20|Total recipe cost per day (price/kg × grams_per_day)|Economically relevant for owner.|
|**Protein source diversity**|0.15|1 - (max_animal_frequency / total_animal_ingredients)|Penalizes recipes repeating same animal in all slots. Relevant because real batch variability that `bioavailability_factors` document (ex: 400% retinol variation in liver) is diluted when multiple species contribute.|
|**Batch robustness**|0.10|Weighted average of inverse `bioavailability_factors` ranges (min/max) of used ingredients|Recipes with lower batch variability ingredients are more predictable.|

**Additional recommended criteria (can be added without architectural change):**

|Criterion (suggested)|Suggested weight|Rationale|
|---|---|---|
|**Simplicity**|0.05|Recipes with fewer ingredients are easier to execute and less prone to human preparation error.|
|**Commercial availability**|0.05|Penalize ingredients hard to find in conventional butchers (ex: bovine spleen, green tripe).|

## 6. Build Pipeline — V10

### 6.1 Overview

`python build_pipeline.py` is the **only** executable script in the system. It is a thin CLI wrapper that delegates to `src/gsd/cli.py:main()`. It operates in **four** modes:

|Mode|When it runs|What it does|
|---|---|---|
|`--generate-mapa`|On demand|Generates MAPA_COMPLETO_JSONs_GSD_Diet_Calc.md (17 sections) from 11 JSONs|
|`--gate-mapa`|On demand|Validates generated MAPA against 8 checks (phantom tokens, counts, divergences)|
|`--audit-mapa`|On demand|Builds CrossRefIndex + validates existing MAPA (drift report)|
|`--validate-db`|On demand|6 assertions (§6.4a a-f) on DB + JSONs|
|`--runtime`|Each user calculation|Reads JSONs → validates → transforms → computes DER/envelope → builds matrix → calls solver → returns result|
|`--build-recipes`|Offline, manual or CI/CD|Generates combinatorial space → runs solver for each combination → ranks → saves `recipes_precomputed.json`|

### 6.2 Detailed Flow (Runtime Mode)

```
1. READ
   ├── DB_ingredientes.json (20 ingredients — 3 supplements still PLANNED, see sat_dados_schema:§9.1)
   ├── constraints.json (60 constraints, solve_cascade ref)
   ├── formulation_rules.json (templates, inclusion, bioavailability, nutrient_matrix)
   ├── audit_provenance.json (refs, flags)
   ├── growth_energy_skeletal.json (Gompertz, TER, DER, k_multipliers)
   ├── objective_weights.json (29 weights, gonadal multipliers)
   ├── scenarios.json (2 scenarios, 17 targets each)
   ├── toxicological_limits.json (8 SULs)
   ├── lp_parameters_schema.json (NUTRIENT_REGISTRY, solve_cascade)
   └── animal_data (sex, weight, height, age, gonadal_status, selection)

2. VALIDATE (input)
   ├── a) every ingredient has exactly 41 nutrients covering nutrients + coverage_excluded_nutrients
   ├── b) every non-USDA source_ref exists in provenance.references
   ├── c) each ingredient category is one of 16+1 from schema enum (now includes 'supplement')
   ├── d) every ingredient_id in category_to_ingredient_mapping exists in DB
   ├── e) NUTRIENT_REGISTRY has entry for all 41 matrix nutrients
   └── f) solve_cascade has at least 1 level with relax_tiers = []

3. COMPUTE DER AND ENVELOPE
   ├── a) Get BW(t) from Gompertz (or from informed weight, if adult)
   ├── b) TER = 70 × BW^0.75
   ├── c) DER = TER × k_multiplier (per scenario and gonadal_status)
   ├── d) minTotal_g = DER / max_energy_density × 0.9
   └── e) maxTotal_g = DER / min_energy_density × 1.1

4. TRANSFORM — as_fed/100g → energy_normalized/1000kcal conversion
   ├── 11 unit conversions (mg→g, ug→mg)
   ├── 2 composite amino acids (Met+Cys, Phe+Tyr) with real values, no proxy
   ├── Apply bioavailability_factors per ingredient
   └── Wildcard expansion (_all_muscle_meat, _all_fat_source) → real ingredient_ids

5. BUILD MATRIX AND CONSTRAINTS
   ├── a_ij: N ingredients × 41 nutrients (energy_normalized/1000kcal)
   ├── Hard constraints (antagonisms, SULs, inclusion) — remain at all levels
   ├── Soft constraints (nutrient_bounds, envelope) — relaxable per solve_cascade
   ├── V10.3: Clinical floor minimum x_i ≥ x_min_i (Level 3 only) — defaults per category
   └── Objective function: Min Σ w_j × (d_j⁻ + d_j⁺)  [w_j includes gonadal multipliers]

6. SOLVE — Declarative Cascade
   ├── FOR level IN solve_cascade:
   │   ├── Build LP: hard + soft constraints (level's relax_tiers → slack)
   │   ├── [V10.3] If level 3: inject x_i ≥ x_min_i (clinical floor) into bounds
   │   ├── Execute solver → raw_result (x_values + nutrient_values)
   │   ├── [V10.3] If level 3 AND infeasible with floor: relax x_min_i → 0, re-solver
   │   ├── IF feasible:
   │   │   ├── Assign result_status and feeding_recommendation from level
   │   │   ├── IF level 1/2: allocations = format_allocations(raw_result) (recommended grams)
   │   │   ├── IF level 3: allocations = null, diagnostic_analysis = build_diagnostic_analysis(raw_result, clinical_floor_info)
   │   │   ├── Compute nutrient_results, gaps, alerts, recommended_additions
   │   │   └── BREAK (do not descend further)
   │   └── ELSE: continue to next level
   └── Level 3 ALWAYS returns result (minimizes violation) — allocations is null

7. VALIDATE (output)
   ├── a) Every ingredient_id referenced in allocations exists in DB (if allocations not null)
   ├── b) Every nutrient_id from 41-matrix has match in nutrient_results
   ├── c) solver_status is one of 3 canonical values
   ├── d) If solver_status = "unsafe_diagnostic", allocations is null AND diagnostic_analysis is present
   ├── e) feeding_recommendation matches solver_status (SAFE_TO_FEED/FEED_WITH_CAUTION/DO_NOT_FEED)
   ├── f) If solver_status ∈ ["optimal", "suboptimal"], allocations not null AND sum of allocations[].grams_per_day within calculated envelope
   ├── g) [V10.2] If unsafe_diagnostic → weight hierarchy validated (hierarchy_valid = true)
   ├── h) [V10.3] If unsafe_diagnostic → clinical_floor_applied ∈ solver_metadata AND clinical_floor_bounds ∈ solver_metadata
   └── i) [V10.3] If clinical_floor_relaxed = true → clinical_floor_relaxation_note MUST exist in what_would_happen

8. WRITE — output as data contract (§7)
```

### 6.3 Detailed Flow (Build-Recipes Mode)

```
1. READ (same 9 JSONs)

2. GENERATE COMBINATORIAL SPACE
   ├── For each diet_template in formulation_rules.diet_templates:
   │   ├── Identify template "slots" (muscle_meat, organ_secreting, bone, etc.)
   │   ├── For each slot, list eligible ingredients (from DB, filtered by category)
   │   └── Generate combinations (C(n,k) per slot, cartesian product between slots)
   ├── Apply pruning: eliminate combinations without minimum coverage (muscle + organ + Ca)
   └── Combinatorial limit: max 1000 combinations per template (configurable)

3. FOR EACH COMBINATION
   ├── Run solver in runtime mode (steps 3-6 above)
   ├── If solver_status = "optimal" AND provenance is complete AND inclusion sums to 100%:
   │   ├── Compute scores for 5 ranking criteria
   │   └── Add to recipe pool
   └── If solver_status = "unsafe_diagnostic": discard (never enters JSON — allocations is null, not a recipe)

4. RANK by rank_composite (§5.2)

5. SAVE versioned recipes_precomputed.json
```


### 6.5 Base Conversion Notes

`valor_1000kcal = valor_100g_as_fed × unit_factor × (1000 / ME_kcal_per_100g)`. This is different from simply changing mg→g — it reindexes from "per 100g of food" to "per 1000kcal of metabolizable energy", which requires computing ME of each ingredient individually via modified Atwater. DB **does not have** `fiber_g` — assumed 0 in calculation; if a vegetable ingredient is added in future, this must be corrected first.

## ✅ Definition of Done — sat_pipeline_fluxo

Conceptual flow and categories are correctly implemented when:

- [ ] Free Mode processes any selection of 1 to N ingredients without pre-solver blocking.
- [ ] Precomputed Recipes Mode reads `recipes_precomputed.json` (generated offline by `python build_pipeline.py --build-recipes`).
- [ ] `recipes_precomputed.json` NEVER contains recipe with `solver_status: "unsafe_diagnostic"` (exclusion rule §5.2).
- [ ] Every recipe has `solver_status: "optimal"`, complete provenance, and inclusion percentages that sum to 100% within declared rounding tolerance.
- [ ] Each recipe in `recipes_precomputed.json` has at least 1 `muscle_meat` + 1 `organ_secreting` + 1 calcium source.
- [ ] `recommended_additions` is generated by category (`ingredient['category']` from DB), mapping `gap.nutrient_id → category(s) with highest concentration`, ordered by real concentration.
- [ ] Contextual alerts follow table in §5.1 (danger for SUL excess/no calcium, warning for deficiency/Ca:P).
- [ ] Build pipeline in runtime mode follows steps 1-8 of §6.2 (READ → VALIDATE → COMPUTE DER/ENVELOPE → TRANSFORM → BUILD MATRIX → SOLVE → VALIDATE OUTPUT → WRITE).
- [ ] `as_fed/100g → energy_normalized/1000kcal` conversion follows formula in §6.5: `valor_1000kcal = valor_100g × fator_unidade × (1000 / EM_kcal_por_100g)`.
- [ ] §A tests (Precomputed Recipes) pass.
